## Where We Left Off

The active frontier is no longer the earlier `SRCH` / request-promotion split.
Those experiments were still useful, but the current highest-signal blocker is
later and more specific:

- native COMINT can now reach `TTYIN ($A072)` on the staged `AMOSL.INI` line
- the post-`A072` editor path then corrupts that line before real command
  execution can proceed

What is proven now:

- native ROM boot still runs through the normal `cpu.step()` / `bus.tick()`
  path
- native boot still performs real SASI-backed reads, but never beyond LBA
  `3326`
- `AMOSL.MON` is still loaded and reached natively
- the observed native LED sequence is still `06, 0B, 00, 0E, 0F, 00`
- the earlier causal edges are still real:
  - `D.REC` at `$0050E4/$0050FC`
  - `JCB+$00` bit 7 at `SCNMOD ($003932)`
  - TCB bookkeeping that makes the staged line visible to COMINT
- with the corrected long-sized `TCB+$12` seed, the first faithful stop is now
  earlier than the old editor-path repro:
  - `A03E` at `$006C10`
  - then `$0011DE -> $001718 -> $001202`
  - `JCB+$00 = $0010`
  - `desc+12 = $000000`
  - no observed `TTYIN` consume yet
  - no timer-suppression hit yet
- a matching word-seed repro reaches the same first `$001202` stop with the
  same state, so the first word-vs-long divergence is later than that shared
  scheduler frontier
- with those seeds in place, native reaches real `TTYIN ($A072)` on the staged
  `AMOSL.INI` buffer
- suppressing timer delivery at `$001E2C/$001E2E` allows 11 native `TTYIN`
  entries, so the first `$000FA2` trip is secondary noise, not the primary
  blocker
- the post-`A072` path `$001E2E..$001E4E -> $00295C/$00299C` rewrites the TCB
  buffer in place
- the staged line visibly mutates:
  - `AMOSL.INI`
  - `AOSL.INI`
  - `OAL.INI`
  - and later an interleaved-zero form seen at the second native `SCNMOD`
- `TCB+$1E` does not naturally advance from `$3E8086` on the native path
- transient `A6=$3F8086` is already live through the post-`A072` editor path

What is still not true:

- native boot still never issues disk reads beyond LBA `3326`
- `AMOSL.INI` on this image is still at LBA `3335`
- native boot still does not reach a real `TRMDEF`-driven terminal session
- `tests/integration/test_boot_native_amosl_ini.py` is still the expected
  xfail

So the scheduler is dequeuing the forced self-link from `$70B0`, not a zero.
Then `$001230` clears that head slot and advances to `$70B4`, where the next
long is zero. That explains the repeated tracer restoration at `$001232`: with
only the self-link present, the queue is a one-element list and native code is
legitimately popping it empty.

A watched write range over `$70B0-$70B7` confirms there is no hidden enqueue
behind that head:

- before the first `$001232` stop, there are no native writes to `$70B4-$70B7`
- the only watched native write is the clear of `$70B0-$70B3` at `$001232`
- the tracer then immediately restores `$70B0` to `$7038`

So the next missing state is not in the dequeue logic at `$00122C/$001230`.
It is earlier: native code never populates the next queue/link slot behind the
forced self-link.

This is now directly proven by the watched queue window:

- at the first `$00122C` stop, `A3=$70B0` and `A3 bytes` start with

- at the first `$001232` stop, the only watched native writes in `$70B0-$70B7`

- there are still no native writes to `$70B4-$70B7`

So native dequeue is behaving consistently: it pops the one forced head entry
and finds no successor. The actual missing producer state is whatever should
have populated the next queue link before the first dequeue.

Refinement from a later stop at `$001238`: the producer does run immediately
after `$001232`, but it still enqueues a null successor.

- `$001234` writes `A6` into `$70B4-$70B7`
- `$001236` writes `A7` into `$70B8-$70BB`

first `$001232` stop. The sharper conclusion is that the queue producer is
executing, but the successor pointer source is already zero by `$001234`.

A narrow forced-register test now confirms that causal edge directly. With
`A6` forced to `$001718` at `$001234`:

- the watched writes become `$70B4-$70B7 = 00 00 18 17`
- `$70B8-$70BB` still receive the same stack value `00 00 1E 76`

So the null successor is not caused by the queue write mechanism itself. It is
caused by `A6` already being zero at `$001234`. That makes the next likely root
cause the preceding `$001232` `MOVE USP,A6` step, or whatever state should have
made `USP` nonzero there.

That intervention is also strong enough to change later scheduler behavior. With

- `A3 = $70B0`

queue/scheduler evolution enough to reach a second dequeue. But it is still not

reach the later `55AA/56BC/56D2` native miss path.

### 2026-03-20 — `SYSTEM@0400` gate decoded

The next useful trace pass made the `$003EBE` gate exact rather than inferred.

- First, the provenance of `SYSTEM=$00302404` is now explicit on the clean
  native path:
  - ROM/monitor copy loops seed `$0400` from the file image as `$00300400`
    (`$03286E`, then `$033184`)
  - `$032560` sets bit `$00000004`
  - `$00ED5C` sets bit `$00002000`
  - `$00F13C` sets bit `$00001000`
  - `$00F160` immediately clears that same `$00001000` bit again
  - **no native write before the gate sets `$00008000`**

- Second, the gate itself decodes to:
  - `$003EBE`: `MOVE.L ($0564).W,D7`
  - `$003EC2`: `ANDI.L #$00000008,D7`
  - `$003EC8`: `BNE $003F7A`
  - `$003ECC`: `MOVE.L ($0400).W,D7`
  - `$003ED0`: `ANDI.L #$0000A000,D7`
  - `$003ED6`: `BEQ $003F7A`
  - `$003EDA`: `ANDI.L #$00008000,D7`
  - `$003EE0`: `BEQ $003F7A`
  - then and only then does the code fall through to `A04C` and the later
    service logic

- So the current live branch at the first hit is not ambiguous anymore:
  - `($0564).W = 1` does **not** trigger the RTC/date path by itself
  - `($0400).W = $00302404` satisfies the `$A000` mask because bit `$2000` is
    set
  - but it fails the `$8000` mask
  - therefore the branch to `$003F7A` is expected under the current emulated
    state

- Third, comparison against generated monitor variants matters but does not
  remove the `$8000` requirement:
  - `AMS4.MON` and `TEST4.MON` differ from `AMOSL.MON` around
    `$003EBE/$003EE4`
  - they use a different pre-test on `($0564).W` and later inspect `($0574).W`
    instead of `($0400).W` at one later point
  - **but they still keep the same `($0400) & $A000` and `($0400) & $8000`
    tests at `$003ECC/$003EDA`**
  - so the missing `$8000` transition is not unique to `AMOSL.MON`

- Fourth, a one-bit experiment proved this state matters:
  - forcing `SYSTEM |= $00008000` exactly once at the first `$003EBE` hit
    changes control flow immediately
  - the run leaves the old `$003EBE -> $003F7A` low-memory loop and moves into
    the `$0043BC-$0043CC` family instead
  - by `5.5M` instructions that patched run still had **not** reached
    `SCNMOD/TTYLIN`, `A06C`, or a live descriptor at `$007074`

Current best statement after this pass:

- the missing `SYSTEM` bit `$8000` is a real missing state transition on the
  emulated path
- but it is not the whole boot failure; setting it moves the machine forward
  but does not complete the transition into higher-level boot work

### 2026-03-20 — upstream `D1` selector and missing `$FFFE40` device

The next pass tightened both sides of the `$8000` story.

- First, the monitor really does contain native code to set
  `SYSTEM |= $00008000`, but the current run never takes it.

  The decision point is around `$00ED40`:

  - live state at the branch ladder:
    - `D1 = $00000004`
    - `SYSTEM = $00300404` before the update
  - the code is a `BTST` ladder on `D1`
  - with bit 2 set, the run takes the `$00ED5C` path
  - that path sets only `SYSTEM |= $00002000`
  - the `SYSTEM |= $00008000` writes at `$00ED74` and `$00ED8C` exist in the
    image, but they are only reached from the `D1` bit 3 / bit 4 branches

  So the current problem is not “there is no `$8000` writer in `AMOSL.MON`”.
  The problem is that the emulated path reaches the wrong `D1` case before the
  `$8000` writer would run.

- Second, once `$8000` is forced, the next missing dependency is concrete.

  The new path enters a subroutine at `$0043BC-$0043D4`:

  - `A5 = $FFFFFE40`
  - the code writes `#1` to `26(A5) = $FFFE5A`
  - reads `26(A5)` back into `D4`
  - tests bit 1
  - if bit 1 is still set, it writes `#0` and loops

  On the current emulator:

  - there is **no** device mapped at `$FFFE40-$FFFE5F`
  - reads from `$FFFE5A` therefore return open-bus `$FF`
  - so the `$0043BC` loop spins forever by construction

- Third, the surrounding code strongly suggests `$FFFE40-$FFFE5F` is not a
  random status block but a second clock/housekeeping register bank.

  After the `$FFFE5A` handshake, the path reads BCD-looking nibbles from:

  - `$FFFE4C/$FFFE4E`
  - `$FFFE50/$FFFE52`
  - `$FFFE54/$FFFE56`
  - `$FFFE58`

  and combines them into decimal values in the same general style as the
  already-known RTC path at `$FFFE04`.

- Fourth, a one-off stub proves the dependency is real.

  With two temporary changes in a probe run:

  - force `SYSTEM |= $00008000` once at the first `$003EBE`
  - register a minimal stub device at `$FFFE40-$FFFE5F` that:
    - returns a handshake byte with bit 1 clear at `$FFFE5A`
    - returns plausible BCD digits at the clock-like offsets

  the machine:

  - gets past `$0043BC`
  - follows the `$FFFE40` path
  - reaches a real native `A06C` at `PC=$0078E4` almost immediately

  That probe still did **not** get as far as a live descriptor at `$007074` by
  `7M` instructions, so the `$FFFE40` block is not the whole boot solution.
  But it is now a proven hardware dependency on the higher path.

Current best statement after this pass:

- the missing `$8000` state is upstream of the first wrong branch, but it is
  selected by `D1`, not by missing code
- the next path after `$8000` depends on a real hardware-visible register bank
  at `$FFFE40-$FFFE5F` that the emulator does not currently model
