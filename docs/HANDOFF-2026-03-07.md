## Where We Left Off

The current committed stopping point is the documentation and integration-test
baseline around native boot and serial-path separation:

- [docs/BOOT-SERIAL-FINDINGS.md](./BOOT-SERIAL-FINDINGS.md)
- [docs/SELF-TEST.md](./SELF-TEST.md)
- [am1200_emulator_spec.md](../am1200_emulator_spec.md)

What is proven on this branch:

- Native ROM boot runs through the normal `cpu.step()` / `bus.tick()` path.
- The ROM performs real SASI-backed disk reads.
- `AMOSL.MON` is loaded by the ROM/SASI path and execution is handed off into
  loaded monitor code.
- The observed native LED sequence is `06, 0B, 00, 0E, 0F, 00`.
- The native handoff test reaches post-sizing `PC = 0x00811C`.
- ROM self-test `LED=5B` uses the main serial block at `0xFFFE20`,
  `0xFFFE24`, `0xFFFE30`, plus `0xFFFE28`.
- The self-test does not use the low-memory `0xFFFFC8/0xFFFFC9` alias path.
- After a matched space on one main port, the first validated self-test output
  line is `300 baud detected\r\n`, and it stays on the matched port.

What is still not proven:

- Native boot does not yet reach `AMOSL.INI` before later low-memory
  serial/interface activity begins.
- Native boot does not yet prove a prompt or a valid `TRMDEF`-driven terminal
  session.
- `0xFFFE28` is still unresolved hardware.
- `0xFFFFC8/0xFFFFC9` should still be treated as a provisional low-memory
  serial/interface path, not as a confirmed alias of primary ACIA port 0.

## Active Investigation Thread

The newest untracked scratch work suggests the next line of investigation had
shifted from self-test work toward monitor command execution and COMINT flow:

- `trace_scnmod.py`
- `patch_driver_v7.py`

`trace_scnmod.py` is specifically aimed at tracing `SCNMOD ($A01C)` while
COMINT processes `SYSTEM SYSMSG.USA`.

Questions that script is trying to answer:

1. Where is the `$A01C` handler?
2. Does it call `SRCH ($1C30)`?
3. Is `SYSTEM` a built-in command or a loadable module?
4. What code path produces `Message #68` ("unrecognized command")?

`trace_scnmod.py` explicitly reuses the boot/patch infrastructure from
`patch_driver_v7.py`, so those two files are the current scratch starting point
if the next app session is continuing the same thread.

## Open Questions

2. What hardware function does `0xFFFE28` perform during self-test setup?
3. What caller enters the low-memory `0x006B68` path, and is it monitor init,
   diagnostics, or an error/report path?
